   - 登录接口正常（200 OK）

3. **问题根因**
   - 前端代码中的 API 请求使用相对路径 `/api/...`
   - 在 `/admin/` 路径下，浏览器会请求 `/admin/api/...`
   - Nginx 的 `/admin/` 规则将所有 `/admin/` 下的请求都转发到前端服务（端口 3000）
   - 前端服务没有 `/api/` 端点，导致 404

---

## 修复方案

### Nginx 配置修改

**原配置**:
```nginx
#PROXY-START/admin/
location ^~ /admin/
{
    proxy_pass http://127.0.0.1:3000/;
    # ...
}
#PROXY-END/
```

**修复后配置**:
```nginx
# API 反向代理 - 必须在 /admin/ 之前
location ^~ /admin/api/
{
    proxy_pass http://127.0.0.1:3001/api/;
    proxy_set_header Host 127.0.0.1;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header REMOTE-HOST $remote_addr;
    proxy_http_version 1.1;
    add_header Cache-Control no-cache;
}

#PROXY-START/admin/
location ^~ /admin/
{
    proxy_pass http://127.0.0.1:3000/;
    # ...
}
#PROXY-END/
```

### 关键点

1. **规则顺序**: `/admin/api/` 必须在 `/admin/` 之前，确保优先匹配
2. **proxy_pass 路径**: `http://127.0.0.1:3001/api/` 会将 `/admin/api/...` 重写为 `/api/...`
3. **API 端口**: 后端服务运行在端口 3001

## 经验总结

### 问题定位经验
1. **浏览器控制台是最直接的调试工具**
   - 能快速定位 404 错误
   - 能看到请求的完整 URL
   - 能看到响应状态和内容

2. **curl 是强大的验证工具**
   - 可以快速验证 API 是否正常
   - 可以测试不同的请求路径
   - 可以查看 HTTP 响应头

3. **Nginx 规则顺序很重要**
   - 更具体的规则必须在通配规则之前
   - `^~` 前缀表示前缀匹配，优先级高于正则匹配

### 部署最佳实践

1. **子路径部署配置**
   - Vite `base: '/admin/'`
   - BrowserRouter `basename="/admin"`
   - Nginx `/admin/api/` 和 `/admin/` 反向代理
   - 三者必须保持一致

2. **API 代理配置**
   - 子路径部署时，API 也需要配置反向代理
   - 规则顺序：API 规则在前，静态资源规则在后
   - 正确设置 `proxy_set_header`

3. **缓存策略**
   - 静态资源：`expires 1m` 或更长
   - API：`Cache-Control no-cache`
   - HTML：`Cache-Control no-cache`

- 添加监控告警，及时发现类似问题
